# StereoPilot LoRA Training Guide

## Overview

This guide explains how to use LoRA (Low-Rank Adaptation) training for StereoPilot. LoRA enables efficient fine-tuning with **99%+ fewer trainable parameters** compared to full fine-tuning.

## Key Benefits

| Aspect | Full Fine-tuning | LoRA Fine-tuning |
|--------|------------------|------------------|
| Trainable Parameters | ~3B | ~3M-30M (0.1%-1%) |
| Checkpoint Size | ~6 GB | ~5-50 MB |
| GPU Memory | 35-40 GB | 25-30 GB |
| Training Speed | Baseline | 1.5-2x faster |
| Preserves Original Model | ❌ | ✅ |

## Quick Start

### 1. Basic LoRA Training

```bash
cd D:\VSCode_MyCode\StereoPilot\SP

python train_lora.py \
    --config toml/infer.toml \
    --train_dir ../SP_Data/mono_train \
    --output_dir ../SP_Data/checkpoints_lora \
    --num_frames 21 \
    --batch_size 1 \
    --lora_rank 4
```

### 2. High-Rank LoRA for Better Adaptation

If you need better adaptation to your dataset, increase the rank:

```bash
python train_lora.py \
    --config toml/infer.toml \
    --train_dir ../SP_Data/mono_train \
    --output_dir ../SP_Data/checkpoints_lora \
    --num_frames 21 \
    --batch_size 1 \
    --lora_rank 16 \
    --epochs 20
```

### 3. Resume Training from Checkpoint

```bash
python train_lora.py \
    --config toml/infer.toml \
    --train_dir ../SP_Data/mono_train \
    --output_dir ../SP_Data/checkpoints_lora \
    --resume_from_checkpoint ../SP_Data/checkpoints_lora/checkpoint_epoch_005.pt
```

## LoRA Parameters Explained

### `--lora_rank` (Default: 4)
- Controls the rank of LoRA decomposition
- **Range**: 1-64 (powers of 2 recommended: 4, 8, 16, 32)
- **Effect**: Higher rank = more expressive power, but more parameters
- **Recommendation**: 
  - Start with rank 4-8 for quick experiments
  - Use rank 16-32 for best performance on large datasets

### `--lora_alpha` (Default: same as rank)
- Scaling factor for LoRA weights
- **Effect**: Controls how much LoRA influences the model
- **Formula**: `scaling = alpha / rank`
- **Recommendation**: Usually set equal to rank

### `--lora_dropout` (Default: 0.0)
- Dropout rate for LoRA layers
- **Range**: 0.0-0.1
- **Recommendation**: 0.0 for small datasets, 0.05-0.1 for large datasets to prevent overfitting

### `--lora_target_modules` (Default: auto)
- Which linear layers to inject LoRA into
- **Default**: `attn.q, attn.k, attn.v, attn.o, ffn.fc1, ffn.fc2`
- **Advanced**: You can specify custom modules if needed

## Memory Optimization

LoRA significantly reduces memory usage. Here's what you can fit on different GPUs:

| GPU VRAM | Full Fine-tuning (max frames) | LoRA Fine-tuning (max frames) |
|----------|-------------------------------|-------------------------------|
| 24 GB | 13 frames | 41 frames |
| 32 GB | 21 frames | 61 frames |
| 40 GB | 21-41 frames | 81 frames |

### Recommended Settings

**For 24GB GPU (RTX 4090):**
```bash
--num_frames 41 --batch_size 1 --lora_rank 4
```

**For 32GB GPU (V100, A100-40GB):**
```bash
--num_frames 61 --batch_size 1 --lora_rank 8
```

**For 40GB+ GPU (A100-40GB, A6000):**
```bash
--num_frames 81 --batch_size 1 --lora_rank 16
```

## Understanding Checkpoints

LoRA training produces two types of files:

### 1. LoRA Weights Only (Small)
- **File**: `lora_weights_epoch_XXX.safetensors`
- **Size**: ~5-50 MB
- **Contains**: Only LoRA parameters (A and B matrices)
- **Use Case**: Share/upload models, fast loading

### 2. Full Checkpoint (Larger)
- **File**: `checkpoint_epoch_XXX.pt`
- **Size**: ~100-500 MB
- **Contains**: LoRA weights + optimizer state + scheduler state
- **Use Case**: Resume training, full reproducibility

### 3. Best Model
- **Files**: `best_lora.safetensors`, `best_checkpoint.pt`
- **Contains**: Best performing model during training

## Monitoring Training

### Loss Curves
Monitor these in TensorBoard:
- `train/loss`: Overall training loss
- `train/mse_loss`: Reconstruction MSE loss
- `epoch/avg_loss`: Average loss per epoch

### Expected Values
- Initial loss: ~0.5-1.0
- Good final loss: < 0.1
- Overfitting warning: Train loss << validation loss

### GPU Memory
Check GPU usage periodically:
```bash
nvidia-smi -l 1
```

Typical memory usage with LoRA:
- Base model: ~20 GB
- LoRA overhead: +2-4 GB
- Batch processing: +3-5 GB
- **Total**: ~25-30 GB

## Inference with LoRA

To use a trained LoRA model for inference:

```python
from models.StereoPilot import StereoPilotPipeline
from lora_utils import LoRAManager
import torch

# Load base model
pipeline = StereoPilotPipeline(config)
pipeline.load_diffusion_model()

# Load LoRA weights
lora_manager = LoRAManager(pipeline.transformer, rank=4)
lora_manager.inject_lora()

# Load trained LoRA
state_dict = torch.load("best_lora.safetensors")
lora_manager.load_lora_state_dict(state_dict)

# Now use pipeline for inference
# ...
```

Or merge LoRA for permanent use:

```python
# Merge LoRA into base weights (irreversible!)
lora_manager.merge_lora_weights()

# Save merged model
safetensors.torch.save_file(
    pipeline.transformer.state_dict(),
    "stereopilot_merged.safetensors"
)
```

## Best Practices

### 1. Start Small
- Begin with `lora_rank=4` and 5-10 epochs
- Check if loss decreases meaningfully
- Increase rank if adaptation seems insufficient

### 2. Monitor Overfitting
- LoRA can overfit quickly on small datasets
- If train loss drops but quality doesn't improve, increase dropout
- Use early stopping based on validation

### 3. Frame Count Selection
- More frames = better temporal consistency
- But also = more memory and slower training
- Good compromise: 21-41 frames for most cases

### 4. Learning Rate
- LoRA typically needs higher LR than full fine-tuning
- Start with 3e-4 (default)
- If loss doesn't decrease, try 1e-3
- If unstable, try 1e-4

### 5. Batch Size
- With LoRA, you can often increase batch size
- But gradient accumulation works just as well
- `batch_size=1` + `gradient_accumulation=4` = effective batch of 4

## Troubleshooting

### Loss Not Decreasing
- **Cause**: Learning rate too low, or rank too low
- **Solution**: Increase LR to 1e-3, or increase rank to 16

### Out of Memory
- **Cause**: Too many frames or batch size too large
- **Solution**: Reduce frames to 21, or use gradient checkpointing

### Poor Quality
- **Cause**: Insufficient training or rank too low
- **Solution**: Train for more epochs, or increase rank to 32

### Slow Training
- **Cause**: Too many frames or I/O bottleneck
- **Solution**: Use 21 frames, increase num_workers to 8

## Comparison with Full Fine-tuning

| Scenario | Full Fine-tuning | LoRA | Winner |
|----------|------------------|------|--------|
| Limited GPU memory | ❌ | ✅ | LoRA |
| Small dataset (<1000 videos) | ❌ | ✅ | LoRA |
| Need original capabilities | ❌ | ✅ | LoRA |
| Maximum quality needed | ✅ | ⚠️ | Full |
| Large dataset (>10000 videos) | ✅ | ⚠️ | Full |
| Quick iteration | ❌ | ✅ | LoRA |

⚠️ = Acceptable performance
